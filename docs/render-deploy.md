# Green Desk — Render.com Deployment Guide

This guide covers deploying Green Desk to [Render.com](https://render.com) using the `render.yaml` Blueprint. It is the fastest path to a live URL — no CLI install required.

## Architecture

```
Internet
  └─► green-desk-web (public, HTTPS, nginx)
        └─► green-desk-api:3001 (private, Render internal network)
              └─► Supabase PostgreSQL (external)
```

The web service is public-facing. The API runs as a **private service** reachable only from within your Render account via the internal hostname `green-desk-api`. The database is Supabase — already provisioned.

---

## Prerequisites

- A [Render.com](https://render.com) account (free tier is enough to start)
- The Green Desk repo pushed to GitHub (or GitLab)
- The Supabase `DATABASE_URL` connection string (from Supabase project → Settings → Database → Connection string → URI mode)

---

## 1. Deploy via Blueprint (recommended)

Render auto-reads `render.yaml` and provisions both services in one click.

1. In Render dashboard → **New → Blueprint**
2. Connect your GitHub repo when prompted
3. Render detects `render.yaml` and previews two services:
   - `green-desk-api` (private service)
   - `green-desk-web` (web service)
4. In the **Environment Variables** panel, fill in the required secret:
   - `DATABASE_URL` → your Supabase connection string
   - Leave `JWT_SECRET` blank — Render generates a secure random value automatically
   - Leave `REDIS_URL` blank for now (see Step 4 if you want Redis)
5. Click **Apply** — Render builds both Docker images and deploys them

The first build takes ~5–8 minutes (Docker layer cache is warm on subsequent deploys).

---

## 2. Find your live URL

Once deployed, the web service URL is shown in the Render dashboard:

```
https://green-desk-web.onrender.com   ← public URL (free tier subdomain)
```

Verify the API is reachable through the proxy:

```bash
curl https://green-desk-web.onrender.com/api/health
# → {"status":"ok"}

curl https://green-desk-web.onrender.com/api/ready
# → {"status":"ready"}
```

> **Free tier note:** Free web services spin down after 15 minutes of inactivity. The first request after a spin-down takes ~30 s. Upgrade to **Starter ($7/mo)** for always-on.

## 3. Set additional secrets

Open the `green-desk-api` service in Render dashboard → **Environment** tab.

### Required

| Variable | Value |
|---|---|
| `DATABASE_URL` | Supabase connection string (set during Blueprint deploy) |
| `JWT_SECRET` | Auto-generated by Render ✓ |

### Stripe billing (optional)

| Variable | Value |
|---|---|
| `STRIPE_SECRET_KEY` | `sk_live_…` |
| `STRIPE_WEBHOOK_SECRET` | `whsec_…` |
| `STRIPE_STARTER_PRICE_ID` | `price_…` |
| `STRIPE_PRO_PRICE_ID` | `price_…` |
| `APP_URL` | `https://green-desk-web.onrender.com` (or your custom domain) |

### Email / SMTP (optional)

| Variable | Value |
|---|---|
| `SMTP_HOST` | e.g. `smtp.sendgrid.net` |
| `SMTP_PORT` | `587` |
| `SMTP_USER` | `apikey` |
| `SMTP_PASS` | SendGrid API key |
| `EMAIL_FROM` | `Green Desk <noreply@example.com>` |

After adding variables, Render triggers an automatic redeploy.

## 4. Redis (optional, recommended for production)

Redis enables shared rate-limiting across API instances and caching of hot reads (floors, desks, billing limits). The API falls back to in-memory operation without it.

**Upstash Redis** is the easiest option:

1. Create a free Redis database at [upstash.com](https://upstash.com)
2. Copy the **REST URL** (starts with `redis://` or `rediss://`)
3. In the `green-desk-api` service → **Environment**, add:
   - `REDIS_URL` → the Upstash connection string

Verify Redis is connected:

```bash
curl https://green-desk-web.onrender.com/api/ready
# With Redis:    {"status":"ready","redis":"ok"}
# Without Redis: {"status":"ready","redis":"unavailable"}
```

## 5. Database migrations

Migrations run automatically on each deploy via the API's startup script. The Render service command (`node dist/migrate.js && node dist/index.js`) runs all pending migrations before serving traffic.

To check migration status manually, use the Render **Shell** tab on the `green-desk-api` service:

```bash
node dist/migrate.js
# → All migrations already applied (or lists newly applied ones)
```

## 6. CI/CD — GitHub Actions

The existing `.github/workflows/ci.yml` deploy job can target Render instead of Fly.io. The Render deploy hook is a URL that triggers a redeploy on POST.

### Set up deploy hooks

For each service in Render dashboard → **Settings → Deploy Hook**, copy the URL.

In your GitHub repo → **Settings → Secrets and variables → Actions**, add:

| Secret | Value |
|---|---|
| `RENDER_DEPLOY_HOOK_API` | Deploy hook URL for `green-desk-api` |
| `RENDER_DEPLOY_HOOK_WEB` | Deploy hook URL for `green-desk-web` |

### Update the deploy job

Replace the `deploy` step in `.github/workflows/ci.yml`:

```yaml
- name: Deploy to Render
  if: github.ref == 'refs/heads/main'
  run: |
    curl -s -X POST "$RENDER_DEPLOY_HOOK_API" &
    curl -s -X POST "$RENDER_DEPLOY_HOOK_WEB" &
    wait
  env:
    RENDER_DEPLOY_HOOK_API: ${{ secrets.RENDER_DEPLOY_HOOK_API }}
    RENDER_DEPLOY_HOOK_WEB: ${{ secrets.RENDER_DEPLOY_HOOK_WEB }}
```

After setup, every push to `main` that passes tests triggers a Render redeploy.

## 7. Custom domain

1. In Render dashboard → `green-desk-web` service → **Settings → Custom Domains** → **Add Custom Domain**
2. Enter your domain (e.g. `app.greendesk.example.com`)
3. Render shows the required DNS records — add them at your DNS provider:

   | Type | Name | Value |
   |---|---|---|
   | CNAME | `app` | `green-desk-web.onrender.com` |

4. Render provisions a Let's Encrypt TLS certificate automatically (~1–2 min after DNS propagates)
5. Update `APP_URL` in the `green-desk-api` environment to your custom domain:
   - `APP_URL` → `https://app.greendesk.example.com`

## 8. Scaling

| Action | Where |
|---|---|
| Upgrade from free to always-on | Service → Settings → Instance Type → Starter |
| Scale to multiple instances | Service → Settings → Scaling → set min/max instances |
| Increase RAM / CPU | Service → Settings → Instance Type |

The API is stateless (Redis handles shared state), so horizontal scaling works without changes.

## Environment variables reference

| Variable | Service | Required | Description |
|---|---|---|---|
| `DATABASE_URL` | API | **yes** | PostgreSQL connection string (Supabase URI) |
| `JWT_SECRET` | API | **yes** | JWT signing secret — auto-generated by Render |
| `JWT_EXPIRES_IN` | API | no | Token lifetime (default: `7d`) |
| `STRIPE_SECRET_KEY` | API | for billing | Stripe secret key |
| `STRIPE_WEBHOOK_SECRET` | API | for billing | Stripe webhook signing secret |
| `STRIPE_STARTER_PRICE_ID` | API | for billing | Stripe price ID for Starter plan |
| `STRIPE_PRO_PRICE_ID` | API | for billing | Stripe price ID for Pro plan |
| `APP_URL` | API | for billing | Public URL used for Stripe redirect and CORS |
| `SMTP_HOST` | API | no | SMTP server hostname |
| `SMTP_PORT` | API | no | SMTP port (default: `587`) |
| `SMTP_USER` | API | no | SMTP username |
| `SMTP_PASS` | API | no | SMTP password |
| `EMAIL_FROM` | API | no | Sender address |
| `CORS_ORIGIN` | API | no | Allowed CORS origin (defaults to `APP_URL`) |
| `REDIS_URL` | API | no | Redis connection string — enables caching and shared rate-limiting |
| `API_HOST` | Web | auto | Set by `render.yaml` to `green-desk-api:3001` — do not change |
| `NODE_ENV` | API | auto | Set to `production` by `render.yaml` |
| `PORT` | API | auto | Set to `3001` by `render.yaml` |
